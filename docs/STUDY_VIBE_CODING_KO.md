# 바이브코딩으로 SnapNote를 만든 과정 — 웹기획자용 스터디

> **대상:** 코드보다 기획·UX·서비스 기획에 가까운 분.  
> **목표:** 이 프로젝트를 **어떤 순서와 개념**으로 만들었는지, 비유와 짧은 정의로 이해하기.

---

## 1. “바이브코딩”이 여기서 뭐야?

**한 줄:** 말(채팅·프롬프트)로 **방향과 디테일을 주고**, 도구(Cursor 등)와 함께 **코드·문서를 빠르게 갈아 끼우며** 완성해 가는 방식에 가깝게 진행한 것을 가리켜요.

**웹 기획에 비유하면:**

- 예전: 시안·스펙만 넘기고 **개발자가 전부 구현**하는 느낌에 가깝고,  
- 바이브코딩에 가깝게 가면: 기획자도 **“이 문장 그대로 스펙에 넣어줘”**, **“이 플로우 E2E로 검증해줘”**처럼 **산출물(문서·테스트·코드)을 같이 밀어** 나갑니다.

SnapNote에서는 **요구사항 문서(SRD/TRD) → 작업 쪼개기(TASKS) → 스프린트별 구현 → E2E/성능/설치 검증** 순으로 **반복**했습니다.

---

## 2. 전체 여정 (큰 그림만)

```
아이디어·초기 방향
    ↓
요구사항·기술 설계 문서 고정 (SRD / TRD)
    ↓
개발 작업 목록으로 쪼개기 (TASKS, 스프린트 S0~S5)
    ↓
한 덩어리씩 구현 + 중간중간 문서/스펙 수정
    ↓
자동 테스트(E2E, 단위, 성능 예산) + Windows 설치 검증
```

**기획자 입장에서 중요한 포인트:**  
처음부터 “완벽한 스펙”이 아니라, **문서가 진실의 원천**이 되게 두고, 구현하면서 **어긋난 부분만 문서와 코드를 같이 고치는** 흐름입니다.

---

## 3. 꼭 알면 좋은 개념 5가지 (쉬운 말)

### ① Electron = “크롬 창 + 진짜 앱 껍데기”

- 화면에 보이는 건 **웹 기술(HTML/React)** 로 만들고,  
- **설치형 프로그램**처럼 쓰게 **포장**한 게 Electron입니다.
- **메인:** 트레이, 단축키, DB 파일, 창 여러 개 관리 — **OS랑 가까운 층**.  
- **렌더러:** 사용자가 보는 **각 창의 UI**.

**비유:** 무대(메인)에서 소품·조명을 돌리고, 관객이 보는 건 스크린(렌더러)에 띄운 화면.

---

### ② IPC = “메인 ↔ 화면 사이 택배”

메인과 렌더러는 **서로 메모리를 공유하지 않습니다.**  
그래서 “메모 목록 줘”, “이 메모 저장해” 같은 건 **IPC**라는 **메시지**로 주고받습니다.

**기획 연결:** “버튼 누르면 저장된다”는 **한 줄**이지만, 뒤에서는 **어느 창이 요청하고 메인이 DB에 쓰는지**가 정리되어 있어야 버그가 줄어듭니다.  
이 프로젝트는 채널 이름을 **상수 파일**로 모아 두어 실수를 줄였습니다.

---

### ③ Preload / `window.snapnote` = “렌더러에 주는 공식 창구”

보안상 **렌더러에 아무 API나 다 열어주면 위험**합니다.  
그래서 **preload**에서 **허용된 기능만** `window.snapnote` 같은 **하나의 이름 아래**에 묶어 줍니다.

**기획 연결:** “앱에서 쓸 수 있는 API 브랜드”를 정한 것과 비슷합니다. 검색·리뷰할 때 `snapnote.`만 찾으면 됩니다.

---

### ④ sql.js = “설치 부담 없는 SQLite”

원래 많은 앱이 **SQLite 네이티브 모듈**을 쓰는데, Windows·Node 버전에 따라 **빌드 지옥**이 나올 수 있습니다.  
그래서 **WASM으로 돌아가는 SQLite(sql.js)** 로 바꿔, **같은 DB 파일**을 쓰되 설치 스트레스를 줄였습니다.

**기획 연결:** “기술 스택 하나 바꿈”이지만, **사용자 입장 경험(로컬 저장, 파일 위치)** 은 유지하는 **트레이드오프 결정** 예시입니다.

---

### ⑤ E2E / 성능 스크립트 = “출시 전 자동 점검표”

- **E2E:** 실제 앱을 띄워 **사용자 플로우**(새 메모 → 접기 → 히스토리 등)를 자동 클릭으로 검증.  
- **성능:** 콜드 스타트·창 열림·히스토리 50개 등 **숫자 목표**를 스크립트로 재실행.  
- **설치 검증:** `build:win` 후 **설치 파일이 제대로 생겼는지** 자동 확인.

**기획 연결:** QA 체크리스트를 **사람만** 하지 않고, **매번 같은 조건으로 돌릴 수 있는 부분**은 자동화해 **회귀(잘 되던 게 안 됨)** 를 잡습니다.

## 4. 웹기획자가 특히 잘 녹여낼 수 있는 부분

| 영역 | 예시 |
|------|------|
| **플로우** | 폴디드 → 미리보기 → 편집 → 다시 접기까지 **끊김 없는지** |
| **카피·레이블** | 버튼·빈 상태·에러 메시지가 **한 제품 톤**인지 |
| **우선순위** | 스프린트마다 “이번엔 히스토리까지, 다음엔 성능”처럼 **TASKS 순서** 조정 |
| **수용 기준** | “검색은 100ms 이내” 같은 NFR을 **측정 가능한 문장**으로 TASK에 적기 |

코드를 직접 짜지 않아도, **문서 한 줄·플로우 한 장**이 바뀌면 구현 방향이 바뀌는 구조라 **기획 영향력이 큽니다.**

## 6. 한 줄로 마무리

**바이브코딩**은 “AI가 다 해준다”가 아니라, **기획·문서·테스트·코드가 같은 방향을 보도록** 대화와 도구로 속도를 내는 방식에 가깝습니다. SnapNote는 그걸 **SRD/TRD/TASKS → 스프린트 → 자동 검증**으로 묶어 둔 작은 사례입니다.
